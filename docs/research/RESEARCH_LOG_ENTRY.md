# Research Log Entry Template

Copy this block to `reports/research_log.md` for every new entry.
Section requirements differ by category — see the per-section guidance below.

---

## YYYY-MM-DD — [Phase XX / Step XX_YY_ZZ] Short descriptive title

**Required for all categories.**

Format the date as ISO-8601. Use `[Phase XX / Step XX_YY_ZZ]` for Category A
work. Use `[CHORE]` for Category C, `[CROSS]` for cross-cutting infrastructure
work, and the relevant chapter code for Category F writing sessions.

**Category:** A (science) | C (chore) | F (writing)

Required for all categories. Exactly one value.

**Dataset:** sc2egset | aoe2companion | aoestats | CROSS

Required for all categories. Use `CROSS` when the entry applies to multiple
datasets or to no specific dataset.

**Artifacts produced:** `path/to/file1.csv`, `path/to/file2.md`

| Category | Requirement |
|----------|-------------|
| A        | Required when the step produces any artifact (notebook, CSV, chart, model file). |
| C        | Optional. Include if the chore creates or modifies tracked output files. |
| F        | Optional. Include draft chapter files or generated tables. |

List paths relative to the repository root. If no artifacts are produced,
omit this line entirely.

---

### What

**Required for all categories.**

One paragraph. State what you did, concretely — actions and outputs, not
motivations. Save the "why" for the next section.

Example (Category A): "Computed ELO deltas for all 1-v-1 sc2egset replays
using a rolling window of 50 prior games per player. Wrote results to
`src/rts_predict/sc2/data/sc2egset/staging/elo_delta.parquet`."

Example (Category C): "Created `docs/templates/research_log_entry_template.yaml`
and added a corresponding human-readable rendering at `docs/research/RESEARCH_LOG_ENTRY.md`."

---

### Why

**Required for all categories.**

One paragraph. State the question or uncertainty you were resolving. Reference
the relevant Scientific Invariant (`.claude/scientific-invariants.md`), phase
definition (`docs/PHASES.md`), or Manual section that motivated this work.
State the hypothesis if there was one.

---

### How (reproducibility)

| Category | Requirement |
|----------|-------------|
| A        | **Required.** Per Invariant #6: a finding without its derivation cannot be cited in the thesis. |
| C        | Optional. Include if a non-obvious command or script was used. |
| F        | Optional. Include if a table or figure was generated by code. |

Paste the literal SQL query or Python snippet that produced the core result.
If the code exceeds approximately 30 lines, link to the committed file and
quote only the key block inline.

```sql
-- Example
SELECT player_id, COUNT(*) AS games_played
FROM matches
WHERE match_time < '2023-01-01'
GROUP BY player_id;
```

---

### Findings

| Category | Requirement |
|----------|-------------|
| A        | **Required.** Every Category A entry must report concrete findings. |
| C        | Optional. Include if the chore surfaces a measurable outcome. |
| F        | Optional. |

Bullet list of factual observations. Numbers must include units. No
interpretation belongs here — save that for "What this means".

- Observation 1: X games dropped due to missing player ID (N = 412, 0.3 % of corpus).
- Observation 2: Median ELO delta = 147 (IQR 64 -- 312).

---

### What this means

| Category | Requirement |
|----------|-------------|
| A        | Include when findings warrant interpretation. |
| C        | Optional. |
| F        | Optional. |

Two or three sentences of interpretation. How does this finding change what
we know about the dataset? Which prior assumption did it confirm or break?
What does it imply for downstream modelling decisions?

---

### Decisions taken

**Required for all categories.**

Bullet list. Each item must include a one-sentence rationale. If no decision
was made, write exactly: "None -- observation only."

- Dropped replays with missing player ID: retaining them would introduce a
  silent null class that corrupts per-player aggregates.
- None -- observation only.

---

### Decisions deferred

| Category | Requirement |
|----------|-------------|
| A        | **Required.** Name the future phase/step where deferred decisions will be resolved. |
| C        | Optional. |
| F        | Optional. |

Describe what must be decided later and at which phase or step that decision
will be made. This section becomes the methodology chapter's "we chose X
because of Y observed in Z" trail of evidence.

Example: "Whether to apply rating-floor filters will be decided in Phase 02
(EDA, Step 02_03 -- rating distribution analysis)."

---

### Acknowledged trade-offs

| Category | Requirement |
|----------|-------------|
| Any      | Include when the entry records a methodology decision (not a pure observation). |

For decision records: what was sacrificed and why it was acceptable.
If no trade-off applies, omit this section entirely.

Example: "Dropping the 0.3 % of replays with missing player IDs reduces
corpus size marginally but eliminates an untraceable null class. The loss is
negligible relative to the risk of silent data corruption."

---

### Thesis mapping

**Required for all categories.**

Name the target section(s) in `thesis/THESIS_STRUCTURE.md` that this entry
feeds. If the entry has no thesis relevance, write: "None -- infrastructure only."

Example: "Section 3.2 (Data Preparation), Section 4.1 (Feature Engineering)."

---

### Open questions / follow-ups

| Category | Requirement |
|----------|-------------|
| A        | **Required.** Do not self-censor. Even "nothing surprising" is a valid signal. |
| C        | Optional. |
| F        | Optional. |

List anything surprising, anything that did not make sense, or anything worth
returning to in a future session. This section is for candid scientific notes,
not polished prose.

- Why do 12 % of ELO deltas exceed 500? Possible de-anonymisation artefact --
  check with raw rating snapshots in Phase 02.
